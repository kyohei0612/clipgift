# これまでの修正内容（中学生向け説明）

「このツールを、もっと壊れにくく・読みやすく・育てやすくする」ために、Claude Code にお願いして整理した内容のまとめです。技術用語を使わずに、**「前はどうだった」→「どう直した」→「なぜ嬉しいのか」**の3点で説明します。

---

## 1. 動く説明書がなかった → 作った

- **前**: フォルダにコードは入っているけど、「これは何？」「どう動かすの？」「どんな命令があるの？」を説明する文書がなかった。
- **直した**: 3つの説明書を置いた。
  - **README.md** — ふつうの使い手向け。「何ができる？」「どうやってインストールする？」
  - **CLAUDE.md** — AI（Claude Code）向け。「このツールの地図」と「踏んだら壊れる地雷の場所」が書いてある
  - **ISSUES.md** — 「今ある問題と直した問題」の一覧
- **嬉しいこと**: 今後このフォルダを初めて見る人（未来の自分も含む）が、いきなりコードを読まなくても全体像をつかめる。

---

## 2. 大きなファイル 1 個に全部書いてあった → 部屋分けした

- **前**: `app.py` という1つのファイルに、1042行もあった。
  - インターネット処理、チャット分析、フォント管理、動画ダウンロード、見張り番…全部ごちゃ混ぜ
  - 料理本で言うと、前菜もメインもデザートも1ページにびっしり書いてある状態
- **直した**: 目的ごとに別ファイルに分けた。
  - `chat_analyzer.py` — チャットを見てハイライト場所を探す係
  - `font_manager.py` — フォントを探す係
  - `system_utils.py` — OS 関係のお世話係（Python の場所とか、ゴミ掃除とか）
  - `paths.py` — フォルダのありかを書くだけの紙
  - `config.py` — 設定値を全部まとめたノート
  - `app.py` は 843 行まで減って、「窓口」に徹するようになった
- **嬉しいこと**: 「チャット分析だけ直したい」時に、その係のファイルだけ見ればいい。どこに何があるか迷わない。

---

## 3. ファイルをアップロードする時に何でも受け取ってた → チェックを入れた

- **前**: 100GB のめちゃくちゃ大きいファイルや、動画じゃないファイル（.exe とか）をアップしても、サーバーが素直に受け取って、途中で転ぶまで気づかない。
- **直した**:
  - 上限 20GB を設定
  - 受け付ける拡張子を決めた（動画は `.mp4 .mkv .mov .webm .m4v`、チャットは `.csv .txt` だけ）
  - 違うものを送られたら「その拡張子はダメです」と即お断り
- **嬉しいこと**: 悪意ある人が変なファイルでサーバーを困らせるのを防ぐ。正しいユーザーにも「それは受け付けないよ」と早く伝わる。

---

## 4. 「今、何してる？」の管理がバラバラだった → 鍵付きの金庫にまとめた

- **前**: クリップ生成中かどうか、今どのクリップを作ってるか、キャンセル要求があるか…などの情報を、あちこちでバラバラに書き換えていた。2人の作業員が同時に書き換えると、どっちの情報が正しいか分からなくなる「喧嘩」が起きる可能性あり。
- **直した**: 「状態ロック（`_state_lock`）」という鍵付きの金庫を作って、状態を書き換える時は必ず鍵を借りてから。
- **嬉しいこと**: キャンセルボタンを押したときに「今どのクリップを作ってたか」が確実に分かる。2つの処理がぶつかって変な状態になる事故を防ぐ。

---

## 5. 見張り番が間違って「誰もいない」と判断することがあった → 判定方法を直した

- **前**: サーバーには「30秒ブラウザから音沙汰なければ自分でシャットダウン」する見張り番が走っている。でも、判定方法が微妙で、クリップ生成中なのに「誰もいないな」と誤解してサーバーを落とす可能性があった。
- **直した**: 「今クリップ処理中」という明確なフラグを作って、見張り番はそれを見るようにした。
- **嬉しいこと**: クリップ生成の途中で勝手にサーバーが死ぬ事故を防ぐ。

---

## 6. キャンセルしてもなかなか止まらなかった → 段階的に強く止めるようにした

- **前**: 「キャンセル」ボタンを押しても、動画エンコードがすぐ止まらないことがあった（`terminate` という「お願い終了」しか送ってなかったので、言うこと聞かないプロセスは無視してた）。
- **直した**: 「お願い終了」で1秒待って、止まらなければ「強制終了」に切り替える2段階方式。
- **嬉しいこと**: キャンセルボタンが「ちゃんと効くボタン」になった。

---

## 7. ログ（記録）が乱雑だった → 整理した

- **前**: `print()` という命令でそのまま画面に出力してた。40箇所以上。エラーも普通の情報も同じ扱いで、後からログファイルを見ても大事なところが埋もれる。
- **直した**: `logger` という正式なログの仕組みに統一。
  - 普通の情報は `INFO`（情報）レベル
  - デバッグ用の細かい情報は `DEBUG`（細部）レベル
  - 警告は `WARNING`、重大なエラーは `ERROR`
  - どのファイルから出たログか（`[chat_analyzer]` とか）が分かるように
- **嬉しいこと**: 問題が起きた時、ログを見て原因を追いやすい。「重大なエラーだけ抽出」もできる。

---

## 8. 文字化けしたチャット CSV が読めなかった → 何度も試すようにした

- **前**: チャットファイル（CSV）は UTF-8 想定で、違う文字コードだと失敗していた。
- **直した**: `UTF-8-BOM → UTF-8 → CP932 → Shift_JIS` の順に試して、どれかで読めたら OK。全部ダメなら最後は不正文字を置換して無理やり読む。
- **嬉しいこと**: Windows の古いソフトで作った CSV も読めるようになった。

---

## 9. 設定値があちこちに散らばっていた → 1冊のノートにまとめた

- **前**: ポート番号は `5000`、アップロード上限は `20 * 1024 * 1024 * 1024`、ハートビート 30秒、再試行 3回…みたいな数字がコードのあちこちに直書きされていた。変更したい時にどこを直せばいいか分からない。
- **直した**: `config.py` という設定ノートに全部集約。しかも環境変数（`CLIPGEN_PORT` など）で、コードを書き換えなくても変更できる。
- **嬉しいこと**: 「ポートを変えたい」「ハートビートを短くしたい」などの時に、1箇所だけ直せば済む。

---

## 10. YouTube からチャットを取るのが「一発勝負」だった → 賢く粘るようにした

- **前**: YouTube のサーバーが「ちょっと混んでます（429 エラー）」と言ってきたら、即失敗して全部やり直し。
- **直した**: 「1 秒待って再試行 → 2 秒 → 4 秒 → 8 秒 → 16 秒」と、間隔を倍々に伸ばして粘る。サーバーが「◯秒待って」と具体的に言ってきたらその秒数を尊重する。
- **嬉しいこと**: ちょっとした通信エラーで全部失敗しなくなる。YouTube にも迷惑をかけない行儀のいい再試行。

---

## 11. ffmpeg が 2 つあってどっち使ってるか謎だった → ルールを決めた

- **前**: 動画を扱う `ffmpeg` というプログラムが、インストーラーが置いた `bin/ffmpeg.exe` と、Python が自動で入れる `imageio_ffmpeg` の 2 つ存在していた。コードのどこでどっちを使ってるか不透明。
- **直した**: `get_ffmpeg_path()` という専用の関数を作って、「まず `bin/` を見る → なければ `imageio_ffmpeg` → なければ PATH から」という優先順位を統一。コード全体で同じルールで呼ぶ。
- **嬉しいこと**: 「どの ffmpeg が動いてるのか」が明確になる。インストール版と開発版で挙動がズレなくなる。

---

## 12. 自動アップデートが途中で失敗すると起動できなくなる可能性があった → 自動で戻すようにした

- **前**: ツールが GitHub から自動更新する途中でネットが切れたりすると、中途半端なファイルが残って次回起動で壊れる可能性があった。
- **直した**:
  - 更新を始める時に「更新中マーカー」という印を置く
  - 無事終わったら印を消す
  - 失敗したらそのまま残す
  - 次回起動時に印が残っていたら「前回失敗したな」と判断して、バックアップ（`.bak`）から自動で元に戻す
- **嬉しいこと**: 更新失敗で起動不能になる事故がなくなる。最悪でも前のバージョンで動く。

---

## 13. マジックナンバー（意味が分からない数字）をなくした

- **前**: コードに `dur = 7.0` とか `if count % 50 == 0` みたいな数字が突然出てきて、何を意味するか読まないと分からない。
- **直した**: それぞれの数字に名前をつけた。
  - `7.0` → `COMMENT_DISPLAY_DURATION_SEC`（コメントを画面に出しておく秒数）
  - `50` → `PROGRESS_LOG_EVERY`（何件ごとに進捗を出すか）
  - `0.08` → `CHAT_BATCH_SLEEP_SEC`（チャット取得の間隔）など
- **嬉しいこと**: コードを読んだ時に「何の数字か」が一目で分かる。変更したい時も名前で検索できる。

## 14. Python のバージョン要件が曖昧だった → はっきり書いた

- **前**: 「Python 3.10 推奨、3.7 以上必須」のような緩い書き方。実行環境によって動いたり動かなかったり。
- **直した**: `pyproject.toml` に `requires-python = ">=3.10"` と正式に明記。README にも明記。
